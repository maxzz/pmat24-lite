## 1. Primary leak: `getDropdownNamesAtom` atom family is never pruned

```6:29:src/renderer/src/store/1-file-mani-atoms/4-cpass-to-login-links/8-reactive-login-names.tsx
export const getDropdownNamesAtom = atomFamily(
    (fileUsCtx: FileUsCtx) => atom(
        (get) => {
            const loginFields = getAllFormsFields_byFileUsCtx(fileUsCtx, { get }).login;
            ...
        }
    )
);
```

`atomFamily` keeps an **internal `Map` holding every atom it ever created**, keyed here by the `fileUsCtx` *object reference*. A fresh `fileUsCtx` object is created every time `createManiAtoms` runs:

```24:25:src/renderer/src/store/0-serve-atoms/0-create/0-create-mani-ctx-atoms/0-create-mani-atoms.ts
        const loginFormCtx = createFormCtx({ fileUs, fileUsAtom, formIdx: FormIdx.login });
        const cpassFormCtx = createFormCtx({ fileUs, fileUsAtom, formIdx: FormIdx.cpass });
```

`createManiAtoms` runs on every file open and on every save/reset. But the dispose path **deliberately keeps `fileUsCtx` alive and never removes it from the family**:

```57:69:src/renderer/src/store/store-utils/1-file-system-utils/8-dispose-file-us.ts
function disposeFormCtx(formAtoms: AnyFormCtx | undefined) {
    if (formAtoms) {
        ...
        //(formAtoms.fileUsCtx as any) = undefined; // Don't clean up file handles
        // Keep `fileUsCtx` intact. Disposing it causes downstream code to crash with
        // `Cannot read properties of undefined (reading 'formIdx')` during folder close/switch.
    }
}
```

Result: each file close/switch/save adds a permanent entry to `getDropdownNamesAtom`'s map. Each entry's closure pins its `fileUsCtx` → `fileUs` (the whole parsed manifest + file content) graph, so heap usage **grows without bound** as the user opens/closes folders and saves manifests. Nothing ever calls `getDropdownNamesAtom.remove(...)` or `setShouldRemove(...)`.

**Fix:** prune the family entry when the form context is disposed, e.g. in `disposeFormCtx`:

```ts
getDropdownNamesAtom.remove(formAtoms.fileUsCtx);
```

(`remove` only drops the cached atom; it doesn't touch the `fileUsCtx` object, so it won't reintroduce the crash that the comment warns about.)

## 4. Latent bug in `atomWithListeners` cleanup (not a growth leak, but can drop a live listener)


If `indexOf` returns `-1` (callback already gone, e.g. StrictMode double-invoke), `prev.slice(0, -1)` removes the **last** listener instead of nothing — corrupting the list. It won't grow memory, but it's fragile. A `if (idx === -1) return prev;` guard fixes it.

## Why this is the right scope — and why I did *not* re-enable the full disposal

I investigated *why* `disposeFileUsManiAtoms` was commented out, and what would actually leak:

1. **What truly leaks on save:** `createManiAtoms` mints a brand-new `fileUsCtx` object every call, and `getDropdownNamesAtom` (an `atomFamily`) caches an atom keyed by that object **forever**. On save the atom is replaced but `disposeFormCtx` never runs, so the *old* `fileUsCtx` family entry is never evicted — and it pins the whole `fileUs` graph. That's the concrete, unbounded leak.

2. **The replaced form atoms themselves are NOT a strong leak.** After `set(fileUs.maniAtomsAtom, newManiAtoms)`, `savedManiAtoms` is only held by the local variable (goes out of scope) and by still-mounted components, which unmount on re-render — jotai then GCs them. The `getDropdownNamesAtom` derived atom reads the *live* atoms via `fileUsCtx.fileUs.maniAtomsAtom` (line 146 of `1-mani-atoms-type.ts`), so it only retains `fileUsCtx`, not the old form atoms. So evicting the family entry is sufficient to stop the growth.

3. **Why re-enabling `disposeFileUsManiAtoms` would risk a new issue:** it runs `discardValues(...)` which nulls the internals of the old `normal`/`manual`/`options` contexts *synchronously*, before React re-renders components that are still mounted on those old contexts. That race is exactly what the codebase's defensive guards (in `loginChangesEffectFn` and the `createManiAtoms` notes) and the `fileUsCtx`-intact comment were added to work around. Re-introducing it is the higher-risk path.

## Confirming no new issues from my chosen fix

- I read the `jotai-family` source: `remove(param)` only `atoms.delete(param)` + notifies listeners. **It does not unmount or destroy the live atom.** So any component still subscribed to `getDropdownNamesAtom(oldCtx)` keeps working; at worst its next render creates a fresh entry (which recomputes the identical value from live data). No crash, no dropped subscription that matters — and after save the component re-renders with the *new* `fileUsCtx` anyway.
- The save branch uses the non-embed `createManiAtoms({ fileUs, fileUsAtom })`, so `savedManiAtoms[0]/[1]` reliably carry the old per-form `fileUsCtx` to evict.
- This path is shared by **reset** too (`4-do-reset-atom` → `updateManiAtomsAfterSaveOrResetAtom`), so reset gets the same cleanup.
- `tsc:web` passes (exit 0) and there are no linter errors. `disposeFileUsManiAtoms` is still used by the close/dispose path, so nothing else broke.

Net effect: the atom-family map no longer grows on save/reset (closing the remaining hole from #1), without taking on the discardValues crash risk.

One small note: I kept the historically-disabled `disposeFileUsManiAtoms`-on-save *off* by design. If you'd eventually like the full disposal back (to also break cross-references for faster GC), that'd be a separate, more involved task — it would need the stale-render race addressed first (e.g. deferring disposal to a microtask/after-commit, or keying the family by a stable primitive). Happy to dig into that if you want, but it's beyond what #2 requires.
